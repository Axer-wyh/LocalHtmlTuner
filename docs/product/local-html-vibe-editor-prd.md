# 本地 HTML Vibe Coding 调优插件 PRD

## 1. 产品概述

### 1.1 产品定位

本产品是一个面向本地 HTML 原型调优阶段的浏览器插件。用户在浏览器预览页面中直接点选或框选目标区域，输入修改意图，插件将页面上下文发送给本地 companion 服务，由本地服务调用已配置的本地 AI 工具 CLI 或 API 模型生成修改方案并应用到真实 HTML/CSS/JS 文件，同时在浏览器中展示修改后的效果、前后对比和可回退版本。

### 1.2 核心价值

在 vibe coding 做产品原型时，用户经常难以准确描述页面中需要调整的位置、元素和交互。本产品通过浏览器中的视觉选择能力，把“我要改这里”转化为 AI 工具可理解的结构化上下文，减少反复描述、误改和低效调优。

### 1.3 MVP 一句话目标

让用户可以在本地静态 HTML 页面预览中选中局部区域，用自然语言描述修改要求，并由本地服务调用已选 AI 工具自动修改本地代码文件，同时支持预览、对比、撤销和回档。

## 2. 目标用户与场景

### 2.1 目标用户

- 使用 AI 进行 vibe coding 的产品经理、设计师、独立开发者。
- 正在用静态 HTML/CSS/JS 快速搭建产品原型的人。
- 不希望手动描述复杂页面结构，但希望精准调优局部视觉和交互的人。

### 2.2 核心使用场景

#### 场景 A：局部视觉调优

用户看到某个按钮、标题、卡片或表单不理想，直接点选该元素，输入“把这个按钮弱化一点，颜色更像次要操作”，本地服务调用已选 AI 工具修改对应 CSS 或 HTML。

#### 场景 B：区域布局调优

用户觉得某一整块布局太挤、卡片之间距离不舒服、内容层级不清晰，框选该区域，输入“这一块整体拉开一点，标题和内容的层级更明显”，AI 工具基于区域上下文生成布局和样式修改。

#### 场景 C：局部交互调优

用户框选某个交互区域，输入“点击这个卡片时增加展开详情的交互”，本地服务调用已选 AI 工具修改 HTML/CSS/JS，并在预览页中验证效果。

#### 场景 D：不满意后快速回退

AI 工具修改后效果不符合预期，用户在插件中点击撤销，回到修改前快照。

## 3. MVP 范围

### 3.1 本版支持

- 仅支持本地静态 HTML/CSS/JS 项目。
- 通过本地 companion 服务启动 `http://localhost:<port>` 预览。
- 支持拖入 HTML 文件或手动绑定本地项目目录。
- 支持浏览器页面内点选单个元素。
- 支持浏览器页面内框选局部区域。
- 支持在 AI 会话工作台输入修改需求。
- 支持本地服务调用用户配置的本地 AI 工具 CLI 或 API 模型修改真实本地文件。
- 支持修改前快照、修改后记录、前后对比、撤销和历史回档。
- 支持基础 Git/GitHub 增强入口，但不作为核心链路前置条件。

## 4. 产品形态

### 4.1 浏览器插件

插件提供页面注入能力和控制面板能力。插件开启后默认在当前浏览器页面右侧打开边栏面板，面板以 AI 会话工作台为主，顶部保留项目信息、目标选择、撤销、对比和设置等功能入口，底部提供向 AI 发送具体修改需求的输入框。用户也可以通过右上角切换图标，将右侧边栏切换为独立于浏览器内容区的小窗口；两种形态共享当前项目、选区、会话、执行进度和输入草稿。

### 4.2 本地 Companion 服务

本地服务运行在用户机器上，负责：

- 管理本地项目目录。
- 启动静态文件预览服务。
- 接收插件传来的选区上下文和修改要求。
- 管理本地快照和回档。
- 检测本机可用 AI 工具 CLI，并根据用户配置调用 CLI 或 API 模型执行代码修改。
- 将执行结果返回插件。

### 4.3 AI 工具执行能力

MVP 阶段需要提供 AI 工具配置和调用能力。优先流程是检测本机已安装的模型 CLI 或代码代理 CLI，例如 Codex CLI、Claude Code、Open Design AMR，再让用户选择使用哪个 CLI、选择模型和推理强度，并测试该 CLI 是否可用。无法或不希望使用本机 CLI 的用户，可以切换到 API/BYOK 模式，配置模型提供商、API Key 和具体模型。

插件不直接写本地文件。文件读取、快照创建、AI 工具调用、补丁应用和回档都由本地 companion 服务完成。本机 CLI 调用优先复用用户本机已登录或已配置的 CLI 环境；API/BYOK 模式下的 API Key 只保存在本地，不写入项目代码、不出现在执行日志中，也不上传到非用户配置的服务。

### 4.4 GitHub 增强能力

GitHub 登录是增强功能，不是使用前提。登录后可用于发布、远程仓库绑定、版本推送和后续 PR 流程。

## 5. 用户流程

### 5.1 首次使用流程

1. 用户安装浏览器插件。
2. 用户启动本地 companion 服务。
3. 用户打开插件面板。
4. 插件检测本机可用 AI 工具 CLI。
5. 用户选择本机 CLI 或 API/BYOK 模式，选择模型和推理强度。
6. 用户测试所选 AI 工具是否可用。
7. 用户拖入 HTML 文件，或选择本地项目目录。
8. 本地服务识别项目目录并启动预览地址。
9. 插件在浏览器中打开 `http://localhost:<port>/<html-file>`。
10. 用户确认当前页面和项目目录绑定关系。
11. 插件进入可用状态。

### 5.2 修改流程

1. 用户在预览页打开插件面板。
2. 用户点击“选择元素”或“框选区域”。
3. 页面进入选择模式。
4. 用户点选元素或拖拽框选区域。
5. 插件显示选区摘要。
6. 用户在 AI 会话工作台底部输入修改要求。
7. 用户发送需求消息。
8. 本地服务创建修改前快照。
9. 本地服务调用已配置的 AI 工具生成修改方案并应用到本地代码。
10. 修改完成后页面刷新。
11. 插件在 AI 会话区显示逐步执行过程、修改结果卡片、文件列表、前后对比和撤销入口。

### 5.3 撤销与回档流程

1. 用户打开插件的版本记录。
2. 用户选择最近一次修改或历史快照。
3. 用户点击“撤销本次修改”或“恢复到此版本”。
4. 本地服务将文件恢复到对应快照。
5. 页面刷新并展示恢复后的效果。

### 5.4 GitHub 增强流程

1. 用户在插件设置中连接 GitHub 账号。
2. 用户绑定或创建远程仓库。
3. 用户将满意的本地版本发布到 GitHub。
4. 插件生成版本说明，并支持后续扩展为分支、commit、push 或 PR。

## 6. 功能需求

### 6.1 项目绑定与预览

#### 需求

- 用户可以拖入 HTML 文件。
- 用户可以手动选择本地项目目录。
- 本地服务根据项目目录启动静态预览服务。
- 插件记录页面 URL 与项目目录的映射。
- 后续打开同一项目的预览 URL 时，插件自动识别项目。

#### 验收标准

- 拖入 `index.html` 后，浏览器可以打开本地服务生成的预览地址。
- 插件面板能显示当前项目目录、入口 HTML 文件和连接状态。
- 关闭再打开页面后，插件仍能识别已绑定项目。

### 6.2 点选元素

#### 需求

- 用户开启点选模式后，页面元素在 hover 时显示高亮边框。
- 点击元素后，插件锁定目标。
- 插件采集目标元素的 DOM 路径、标签名、文本、class、id、尺寸、坐标、计算样式和附近 DOM 上下文。
- 用户可以重新选择或取消选择。

#### 验收标准

- 用户能准确选中按钮、文本、图片、表单项、卡片等常见元素。
- 插件面板能展示简洁的选中目标摘要。
- 选择模式不会永久破坏原页面交互。

### 6.3 框选区域

#### 需求

- 用户开启框选模式后，可以在页面上拖拽出矩形区域。
- 插件根据矩形区域识别覆盖范围内的相关 DOM 节点。
- 插件采集区域截图、坐标、区域内主要元素列表、共同父节点候选和相关样式。
- 用户可以重新框选或取消。

#### 验收标准

- 用户能框选一组卡片、一个内容区、一个表单区或一段交互区域。
- 插件能把区域内主要 DOM 节点传给本地服务。
- AI 工具能基于区域上下文进行局部布局或交互修改。

### 6.4 修改请求输入

#### 需求

- AI 会话工作台底部提供自然语言输入框。
- 用户可以描述视觉、布局、文案或交互调整。
- 工作台顶部展示当前项目、选区、撤销、对比和 AI 工具配置状态。
- 执行进度以内嵌 AI 会话消息和工具调用进度块展示，不使用独立执行进度页面。
- 请求提交前，插件校验本地服务连接、项目绑定状态和 AI 工具可用状态。

#### 验收标准

- 未连接本地服务时，插件给出明确提示。
- 未选择目标时，插件提示用户先选择或框选。
- 未选择可用 AI 工具时，插件提示用户先完成本机 CLI 检测、CLI 选择或 API/BYOK 配置。
- 执行中用户能看到当前阶段，例如“创建快照”“分析上下文”“修改文件”“刷新预览”。

### 6.5 AI 工具 CLI 与模型配置

#### 需求

- 插件或本地服务可以扫描本机 PATH，检测可用 AI 工具 CLI。
- 配置界面需要展示当前使用模式：使用本机 CLI，或使用 API/BYOK。
- 本机 CLI 模式下，需要展示检测到的代码代理列表、版本号、可用状态和当前选中项。
- 用户可以选择具体 CLI、模型和推理强度。
- 用户可以点击“重新扫描 PATH”刷新本机 CLI 检测结果。
- 用户可以发起一次轻量可用性测试，确认所选 CLI 或模型能被调用。
- API/BYOK 模式下，用户可以选择模型提供商、填写 API Key 并选择具体模型。
- API Key 只允许保存在本地安全存储或本地服务配置中。
- API Key 不写入项目目录、不进入快照、不显示在日志中。

#### 验收标准

- 用户完成本机 CLI 选择并测试成功后，插件能显示“已连接”或类似状态。
- 插件能显示当前使用的 CLI 名称、版本、模型和推理强度。
- 未检测到 CLI 时，用户可以重新扫描 PATH，或切换到 API/BYOK 模式。
- API Key 不会出现在项目文件、快照文件、执行摘要或插件普通日志中。
- AI 工具配置缺失或不可用时，用户无法提交修改任务，并能看到明确引导。

### 6.6 AI 会话式修改执行

#### 需求

- 本地服务接收插件提交的结构化任务。
- 任务包含项目目录、入口页面、选区信息、截图信息、用户修改要求和当前文件上下文。
- 本地服务在调用 AI 工具前创建快照。
- 本地服务将任务上下文组织为 CLI 或 API 请求。
- AI 工具需要以会话式过程返回进度，例如分析目标、定位文件、生成补丁、应用修改、刷新预览。
- AI 工具返回修改方案或补丁。
- 本地服务校验修改范围后，将修改应用到本地 HTML/CSS/JS 文件。
- 本地服务返回修改结果、变更文件列表和执行摘要。

#### 验收标准

- AI 工具修改完成后，本地文件实际发生变化。
- 插件工作台能在 AI 会话区展示逐步回复和执行进度，不需要跳转到单独进度页面。
- 插件能展示本次修改涉及的文件。
- 修改失败时，插件能展示失败原因，并保留修改前快照。

### 6.7 预览与对比

#### 需求

- 修改前保存目标区域或整页截图。
- 修改后刷新预览页面。
- 插件支持修改前/修改后切换查看。
- 插件显示本次变更的文件列表和简要说明。

#### 验收标准

- 用户能看到修改前后的视觉差异。
- 刷新不会丢失当前任务结果。
- 对比功能不要求复杂像素 diff，MVP 使用前后截图切换即可。

### 6.8 快照、撤销与回档

#### 需求

- 默认使用本地快照管理版本。
- 每次修改前保存受影响文件的 before 快照。
- 修改后记录 after 状态和 manifest。
- 用户可撤销最近一次修改。
- 用户可恢复到某个历史快照。
- 快照不依赖浏览器缓存。

#### 建议快照结构

```text
.vibe-checkpoints/
  2026-06-09-153012/
    manifest.json
    before/
      index.html
      styles.css
      app.js
    after/
      index.html
      styles.css
      app.js
```

#### 验收标准

- 每次修改前都能生成快照。
- 用户点击撤销后，本地文件恢复到修改前状态。
- 用户恢复历史版本后，预览页面展示对应版本。

### 6.9 Git/GitHub 增强

#### 需求

- 插件可以检测当前目录是否是 Git 仓库。
- 插件可以展示基础 Git 状态。
- 登录 GitHub 后，用户可以绑定远程仓库。
- 用户可以把满意版本发布到 GitHub。

#### MVP 边界

- GitHub 登录不阻塞核心修改流程。
- GitHub 不承担默认回档能力。
- PR、分支策略、协作审阅可放到后续版本。

## 7. 数据与上下文

### 7.1 插件发送给本地服务的数据

- 当前预览 URL。
- 项目 ID 或项目目录映射 ID。
- 选择模式：点选或框选。
- 选中元素 DOM 信息。
- 框选区域坐标和尺寸。
- 区域截图或截图引用。
- 计算样式摘要。
- 相关父子 DOM 结构。
- 用户自然语言修改要求。

### 7.2 本地服务维护的数据

- 项目目录映射。
- 预览服务端口。
- 快照记录。
- 修改任务记录。
- AI 工具 CLI 检测结果。
- 当前 AI 工具配置状态。
- AI 会话式修改执行结果。
- 可选 GitHub 连接状态。

### 7.3 隐私与安全原则

- 本机 CLI 模式下优先在用户本机环境中执行，不额外要求用户输入 API Key。
- 不上传本地代码，除非用户明确配置 API/BYOK 模型服务或 GitHub 发布。
- API Key 只保存在本地安全存储或本地服务配置中，不写入项目文件、快照和普通日志。
- 不在日志中输出 token、密钥、`.env` 值或凭证。
- 文件修改必须发生在用户绑定的项目目录内。
- 插件和本地服务之间应有本地鉴权或配对机制，避免任意网页调用本地服务。

## 8. 技术架构概要

```text
用户操作
  -> 浏览器插件
    -> 页面选择层
    -> 插件控制面板
    -> 上下文采集
  -> 本地 Companion 服务
    -> 项目目录管理
    -> 静态预览服务
    -> 快照/回档管理
    -> 本机 AI CLI 检测
    -> AI 工具配置与调用
    -> 补丁生成与应用
  -> 本地文件系统
    -> HTML/CSS/JS 文件修改
    -> .vibe-checkpoints 快照
  -> 浏览器预览反馈
    -> 刷新页面
    -> 前后截图对比
    -> 确认/撤销/回档
```

## 9. 成功指标

### 9.1 MVP 可用性指标

- 用户能在 3 分钟内完成首次项目绑定和预览启动。
- 用户能在 30 秒内选中目标区域并提交修改要求。
- 常见静态 HTML 页面中，点选和框选能稳定产出可用上下文。
- 修改失败时，用户能通过撤销或回档恢复。

### 9.2 产品效果指标

- 减少用户对页面位置的复杂自然语言描述。
- 降低 AI 工具误改非目标区域的概率。
- 缩短局部 UI 调优的反馈循环。
- 提高用户对 AI 修改结果的可控感。

## 10. 风险与待验证点

### 10.1 技术风险

- 框选区域与源码文件之间的映射可能不稳定。
- AI 工具可能修改过多文件或误解需求。
- 本机 CLI 检测可能受 PATH、安装路径和登录状态影响。
- API Key 本地存储和日志脱敏需要严格处理。
- 插件注入层可能影响原页面交互。
- 本地服务鉴权需要谨慎处理，避免本地文件被恶意网页访问。

### 10.2 产品风险

- 用户可能不理解本地服务和插件的关系。
- 非技术用户可能不理解项目目录、预览服务、快照等概念。
- 如果修改质量不稳定，用户会更依赖撤销，而不是继续使用。

### 10.3 验证重点

- 点选和框选是否真的比自然语言描述更高效。
- AI 工具是否能基于选区上下文准确修改局部代码。
- 快照/撤销是否足够可靠。
- 本地服务启动和项目绑定是否足够顺滑。
